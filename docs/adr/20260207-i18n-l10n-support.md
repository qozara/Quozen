# Architecture Decision Record: Internationalization & Localization

## Context
Quozen aims to expand beyond English-only users, requiring support for multiple languages (initially English and Spanish). This necessitates translating static UI elements and formatting dates, currencies, and numbers according to regional conventions without reloads.

## Decision
1. **Library Standardization:** Use `react-i18next` and `i18next-browser-languagedetector` for the core translation engine.
2. **Language Locales:** Initial support strictly limits to `en-US` and `es-419` (LatAm Spanish). All specific variations (e.g., `es-ES`, `en-GB`) map back to these base locales.
3. **State & Persistence:** Store the override locale preference directly in the `quozen-settings.json` file inside Google Drive, so it follows the user across devices. A fallback of "System" triggers the browser detector.
4. **Dates and Numbers:**
   - Create a localized date formatting wrapper hook (`useDateFormatter`) using `date-fns` locale objects dynamically.
   - For currencies and large numbers, rely on the native `Intl.NumberFormat` API rather than manual string interpolation.
5. **Separation of Concerns:** Translatable strings apply only to static app text. User-generated content (e.g., Expense names, Group names) and raw API backend errors are NOT translated.

## Consequences
- Requires upfront extraction of all strings into JSON dictionaries (`src/locales/en/translation.json`, `src/locales/es/translation.json`).
- Minor performance hit injecting i18n context, mitigated by correct bundling.
- Simplifies UI code by removing manual string interpolation for currencies (`Intl.NumberFormat` takes over).
